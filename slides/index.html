API عمومی

X برای تغییراتی که با نسخه‌های قبل تطابق ندارند
Y برای تغییراتی که با نسخه‌های قبل سازگارند
Z برای باگ‌فیکس
چگونه؟!
نرم‌افزار‌هایی که از نسخه‌بندی معنایی استفاده میکنند باید یک API عمومی تعریف کنند. این API میتواند در خود کد و یا مؤکداً در مستندات موجود باشد. در هر صورت این توضیحات باید جامع و شامل باشند.
یک شماره نسخه صحیح باید به‌ شکل X.Y.Z باشد که در آن Y ،X و Z اعداد صحیح غیر منفی هستند که
X نسخه‌ی Major
Y نسخه‌ی Minor و
Z نسخه‌ی Patch می‌باشد.
هر عنصر باید یک به یک و بصورت عددی افزایش پیدا کند. به عنوان مثال: 1.9.0 -> 1.10.0 -> 1.11.0
هنگامی که شماره نسخه Major یک واحد افزایش می‌یابد، نسخه‌های Minor و Patch باید صفر شوند. همچنین زمانی که شماره نسخه Minor افزایش می‌یابد، نسخه‌ی Patch باید صفر شود. برای مثال: 1.1.3 -> 2.0.0 , 2.1.7 -> 2.2.0
پس از انتشار یک نسخه محتویات آن به هیچ‌وجه نباید تغییر کند. هرگونه تغییر باید به عنوان یک نسخه جدید منتشر شود.
نسخه‌هایی با شماره Major صفر (0.Y.Z)، نسخه‌های درحال توسعه هستند. هرگونه تغییری ممکن است در هر زمان اتفاق بیوفتد؛ پس API عمومی نباید پایدار فرض شود.
نسخه‌ی 1.0.0، API عمومی را تعریف میکند. به اینصورت که افزایش شماره‌های نسخه از این پس به این API و چگونگی تغییر آن بستگی دارد.
شماره نسخه Patch یا Z (x.y.Z به شرطی که x > 0) باید در صورتی که باگ‌فیکس‌های مطابق با نسخه قبل ارائه شدند افزایش یابد. باگ‌فیکس به تغییراتی داخلی اتلاق میشود که رفتار نادرست برنامه را اصلاح میکند.
درصورتی که کاربرد جدیدی که با نسخه‌های قبلی سازگاری دارد توسط API ارائه شود و یا هر یک از کاربردهای قبلی حذف شوند ، باید شماره نسخه Minor یا Y (x.Y.z به شرطی که x > 0) افزایش یابد. همچنین در صورتی که کاربرد‌های جدید خاص یا ارتقاع در کد خصوصی ارائه شده باید میتوان این شماره را افزایش داد. تغییرات مذکور میتواند شامل تغییرات مربوط به نسخه‌ی Patch نیز باشد. شماره نسخه Patch باید پس از افزایش Minor صفر شود.
درصورت ارائه‌ی تغییراتی که با نسخه‌های قبل سازگاری ندارند شماره نسخه‌ی Major یا X (X.y.z به شرطی که X > 0) باید افزایش یابد. تغییرات مذکور میتواند شامل تغییرات مربوط به نسخه‌ی Minor و Patch نیز باشد. شماره‌های Patch و Minor نسخه باید پس از افزایش شماره‌ی Major صفر شوند.
میتوان یک نسخه‌ی pre-release (پیش از انتشار) را با افزودن خط تیره (-) و نمایه‌هایی که با نقطه از هم جدا شده‌اند، دقیقاً بعد از شماره نسخه Patch معرفی کرد. نمایه‌ها باید محدود به کارکتر‌ها و اعداد ASCII و خط تیره (-) باشند. نسخه‌های pre-release کاربردی هستند اما نسبت به شماره نسخه‌ی اصلی از اهمیت کمتری برخوردارند. نمونه‌ها: 1.1.3 -> 2.0.0 , 2.1.7 -> 2.2.0
میتوان یک نسخه‌ی build را با افزودن علامت جمع (+) و مجموعه‌ای از نمایه‌هایی که با نقطه از هم جدا شده‌اند، دقیقاً بعد از شماره نسخه‌ی Patch یا نسخه‌ی pre-release معرفی کرد. نمایه‌ها باید محدود به کارکتر‌ها و اعداد ASCII و خط تیره (-) باشند. نسخه‌های build کاربردی هستند و نسبت به شماره نسخه‌ی اصلی از اهمیت بیشتری برخوردارند.‍ نمونه‌ها: 1.0.0+build.1, 1.3.7+build.11.e0f985a
اهمیت نسخه‌ها باید با جدا کردن شماره نسخه به بخش‌های Major, Minor, Patch, pre-release و build به ترتیب مذکور. نسخه‌های Major, Minor و Patch همیشه به روش عددی مقایسه میشوند. اهمیت نسخه‌های pre-release و build باید با مقایسه‌ی هر بخش جدا شده توسط نقطه به روش زیر تعیین شود:
نمایه‌هایی که فقط از اعداد تشکیل شده‌اند به روش عددی و نمایه‌هایی که از حروف و خط تیره (-) تشکیل شده‌اند به ترتیب لغوی مقایسه میشوند. نمایه‌های عددی همیشه از اهمیت کمتری نسبت به نمایه‌های غیر عددی برخوردارند. نمونه‌ها: 1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta.2 < 1.0.0-beta.11 < 1.0.0-rc.1 < 1.0.0-rc.1+build.1 < 1.0.0 < 1.0.0+0.3.7 < 1.3.7+build < 1.3.7+build.2.b8f12d7 < 1.3.7+build.11.e0f985a
